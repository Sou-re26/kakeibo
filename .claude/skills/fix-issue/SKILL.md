# Fix Issue

CLAUDE.md「既知の課題」から1件を選び、修正→検証→リスト更新までを一貫して行う。
**原則: 1回の実行で1課題。検証を通っていない修正は完了と呼ばない。**

## 手順

### 1. 課題の選定
- CLAUDE.md の「既知の課題」を読む。引数があれば照合して対象を確定する。
- 引数が空なら「バグ(優先度高)」から上位2〜3件を、依存関係(先にやるべき前提)込みで提示して選んでもらう。
- **同根の課題はセットを明示する**(例: 「カテゴリ選択で入力内容が消える」はウィザード状態のContext化とセット。着手前にスコープを確認する)。

### 2. 計画の提示
`git status` を確認し、未コミットの変更が既にあれば扱いを確認する(修正の差分と混ざるのを防ぐ)。着手前に3行以内で: 対象課題 / 修正方針 / 触るファイル。DBスキーマ変更を伴う場合は CLAUDE.md「DBスキーマ変更の手順」(drizzle-kit generate → SQL確認)に従うことを明記し、マイグレーション生成前に確認を取る。

### 3. 最小修正
CLAUDE.md「コーディング規約」を遵守する。特に:
- 色はハードコードせず `constants/theme.ts` の `Colors` へ
- 新規カラム・保存値はコード値(表示用文字列をDBに入れない)
- 金額は円の整数として扱う
- 一覧の更新は `useFocusEffect` か `useLiveQuery`
- プラットフォーム専用APIは分岐かラッパー経由
- 新しいアイコン名は `icon-symbol.tsx` の `MAPPING` にも追記

課題リスト外の問題に気づいても直さない(報告に留める)。

### 4. 検証
- `npm run check`(lint + 型チェック)を実行し、結果を報告に含める。
- 実行時挙動の変更は、可能なら再現手順ベースで確認する(verify Skillの手順)。エミュレータでの確認ができない場合は、確認すべき操作手順を報告に書き「未検証」と明記する。

### 5. CLAUDE.md の更新
- 直した課題をリストから削除する。
- 作業で判明した新しい事実・新たな課題があれば該当セクションに追記する(古い記述は削除)。

### 6. 報告
```markdown
# 課題修正: <課題名>
- 変更: <ファイルと内容を1行ずつ>
- 検証: <npm run check の結果 / 実機確認の結果または「未検証」+確認手順>
- CLAUDE.md: <消した課題・追記した内容>
- 気づき: <直さなかった周辺の問題(あれば)>
```
コミットは依頼された場合のみ行う。

## エッジケース

| 状況 | 対応 |
|---|---|
| 課題の修正が別の課題の前提になっている | 順序を示して前提側から着手することを提案する |
| 修正がスキーマ変更を要する | マイグレーション手順に従い、生成SQLを確認してから適用。**コミット済みマイグレーションは編集しない** |
| 引数の課題がリストにない | リスト外である旨を伝え、リストへの追加(内容の確認)を先に行う |
| 課題の症状は分かるが原因が未診断 | 先に debugging-specialist で原因を特定し、結果を「既知の課題」に追記してから修正に着手する |
| 修正の途中で想定よりスコープが広いと判明 | 中断して現状と選択肢を報告する(黙って大改修に進まない) |

## 禁止事項

- 複数課題の同時修正、リスト外の「ついで修正」
- `npm run check` を通していない完了報告
- CLAUDE.md の課題リストを更新しない完了報告
- 手動での `drizzle-kit push`(マイグレーションは起動時自動適用)
